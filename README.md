# LLM Log Masker

This project provides a Python application to mask proprietary and sensitive information from application log files using a locally hosted Large Language Model (LLM) via Ollama. It identifies sensitive data, replaces it with consistent placeholders, and outputs both the masked logs and a reference map of the masked data.

## Features

-   Identifies a wide range of proprietary information (names, emails, IPs, custom IDs, etc.) using an LLM.
-   Masks identified information with consistent, typed placeholders (e.g., `[EMAIL_ADDRESS_1]`, `[IP_ADDRESS_2]`).
-   Outputs a JSON file containing each original log line, its masked version, and the specific PII extracted from that line.
-   Outputs a global JSON file mapping all unique original PII values to their consistent placeholders.
-   Supports configurable Ollama models.
-   Includes error handling and structured logging.

## Prerequisites

1.  **Python**: Version 3.8 or higher.
2.  **Ollama**: Ensure Ollama is installed and running on your system. You can download it from [ollama.com](https://ollama.com/).
3.  **LLM Model**: You need to have an LLM model pulled into your Ollama instance that is suitable for text analysis and instruction following (e.g., Llama2, Mistral).

## Configuration

1.  **Ensure Ollama Server is Running:**
    Start your Ollama server. Typically, it runs in the background after installation. You can verify by opening `http://localhost:11434` in your browser or using `ollama list` in your terminal.

2.  **Pull an LLM Model:**
    If you don't have a suitable model, pull one using Ollama. For example, to pull Llama2 (a common default):
    ```bash
    ollama pull llama2
    ```
    Other models like `mistral` or more specialized fine-tuned models can also be used. The script defaults to `llama2` but can be changed via a command-line argument.

## Usage

Run the `log_masker.py` script from your terminal, providing the path to the log file you want to process.

**Command-line arguments:**

```
usage: log_masker.py [-h] [--model MODEL] [--output_masked OUTPUT_MASKED]
                     [--output_pii_map OUTPUT_PII_MAP] [--loglevel LOGLEVEL]
                     logfile

Masks proprietary information in log files using an LLM.

positional arguments:
  logfile               Path to the input log file.

options:
  -h, --help            show this help message and exit
  --model MODEL         Name of the Ollama model to use (default: llama2).
  --output_masked OUTPUT_MASKED
                        Path for the output JSON file with masked logs (default: masked_logs_output.json).
  --output_pii_map OUTPUT_PII_MAP
                        Path for the output JSON file with the global PII map (default: global_pii_reference_map.json).
  --loglevel LOGLEVEL   Set the logging level (default: INFO). Choices: DEBUG, INFO, WARNING, ERROR, CRITICAL.
```

**Example command:**

```bash
python log_masker.py /path/to/your/sample.log --model llama2 --loglevel DEBUG
```

This will:
-   Process `/path/to/your/sample.log`.
-   Use the `llama2` model from Ollama.
-   Output masked logs to `masked_logs_output.json` (default name).
-   Output the global PII map to `global_pii_reference_map.json` (default name).
-   Set logging to DEBUG level for verbose output.

## Input Log Format

The script is designed to process generic log files where each line is a separate log entry (plain text). The LLM attempts to identify sensitive information based on context within each line.

**Example `sample.log` line:**

```
User 'markhen@example.com' accessed customer records for account ACC_NEW_456 from IP 10.10.101.10.
```

## Output JSON Formats

Two JSON files are generated by default:

1.  **`masked_logs_output.json`**:
    Contains a list of objects, where each object corresponds to an input log line and includes:
    -   `original_log`: The original log line.
    -   `masked_log`: The log line with identified PII replaced by placeholders.
    -   `pii_extracted_this_log`: A dictionary mapping the placeholders used in this specific masked log line to their original values.

    **Example entry in `masked_logs_output.json`:**
    ```json
    [
      {
        "original_log": "User 'markhen@example.com' accessed customer records for account ACC_NEW_456 from IP 10.10.101.10.",
        "masked_log": "User '[EMAIL_ADDRESS_1]' accessed customer records for account [ACCOUNT_NUMBER_2] from IP [IP_ADDRESS_3].",
        "pii_extracted_this_log": {
          "[EMAIL_ADDRESS_1]": "markhen@example.com",
          "[ACCOUNT_NUMBER_2]": "ACC_NEW_456",
          "[IP_ADDRESS_3]": "10.10.101.10"
        }
      }
      // ... more entries
    ]
    ```

2.  **`global_pii_reference_map.json`**:
    A JSON object that provides a global mapping of all unique original PII values found across all processed log lines to their single, consistent placeholder. This helps in cross-referencing.

    **Example `global_pii_reference_map.json`:**
    ```json
    {
      "markhen@example.com": "[EMAIL_ADDRESS_1]",
      "ACC_NEW_456": "[ACCOUNT_NUMBER_2]",
      "10.10.101.10": "[IP_ADDRESS_3]",
      "jane.doe": "[USERNAME_4]",
      "203.0.113.45": "[IP_ADDRESS_5]"
      // ... all unique PIIs and their global placeholders
    }
    ```

## How it Works (Briefly)

For each log line:
1.  A detailed prompt is constructed, instructing the LLM to identify sensitive data, provide a masked version of the log, and list the original values with their types.
2.  The log line and prompt are sent to the configured Ollama LLM.
3.  The LLM's response (expected to be in a specific structured format) is parsed.
4.  For each piece of PII identified by the LLM:
    a.  A consistent placeholder is generated or retrieved from a global map (e.g., `markhen@example.com` will always map to `[EMAIL_ADDRESS_1]`).
    b.  The original log line is reconstructed with these consistent placeholders.
5.  The results are collected and written to the output JSON files.

## Troubleshooting & Notes

-   **Ollama Server**: Ensure the Ollama server is running and accessible. If the script reports connection errors, this is the first thing to check.
-   **Model Availability**: Make sure the specified model (e.g., `llama2`) has been pulled via `ollama pull <model_name>`.
-   **LLM Performance**: The accuracy of PII detection heavily depends on the capability of the chosen LLM and the effectiveness of the prompt. Results may vary. The prompt within `log_masker.py` can be further refined for better performance with specific models or data types.
-   **Processing Time**: LLM processing can be time-consuming, especially for large log files.
-   **Output Format Adherence**: The script relies on the LLM adhering to the requested output format. While there's parsing logic to handle some deviations, significant deviations might lead to warnings or incomplete masking for some lines. Check logs for warnings.
-   **Memory Usage**: For very large log files, loading all results into memory before writing to JSON might be an issue. For extremely large files, a streaming approach or batch processing might be considered as a future enhancement.
